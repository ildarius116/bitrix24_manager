# Bitrix24 «Рабочий день» (incomsystem)

Знания добыты ресёрчем (read-only). Полная постановка — `PRD.md`, план — `PLAN.md`.
Транспорт и гарды (allowlist, plan→execute, идемпотентность, аудит, ретраи) — берём из
соседнего скилла `bitrix24-agent` (`scripts/bitrix24_client.py`).

## Доступ

- Аутентификация: входящий **вебхук** (scope `crm,task,timeman,user`), креды в `.env` проекта:
  `B24_DOMAIN`, `B24_AUTH_MODE=webhook`, `B24_WEBHOOK_USER_ID=1244`, `B24_WEBHOOK_CODE=<секрет>`.
- Код вебхука — СЕКРЕТ: только в `.env` (вне git), маскировать в логах, после работ перегенерировать.
- user 1244 = Ильдар Сабиров (владелец записей «Рабочий день»).

Клиент (из проекта):
```bash
PY=./venv/Scripts/python.exe
CLIENT=.claude/skills/bitrix24-agent/scripts/bitrix24_client.py
set -a; source .env; set +a
$PY "$CLIENT" user.current --params '{}' --allow-unlisted   # smoke
```

## Метамодель (подтверждено)

### entityTypeId **1208** — «Рабочий день» (categoryId=61, одна запись/день)
| Поле | Тип | Смысл |
|---|---|---|
| `title` | string | «ФИО \| дд.мм.гггг» |
| `ufCrm46_1742342657` | date | **дата рабочего дня** (надёжный источник даты) |
| `ufCrm46_1742997115` | crm[] | **«Работы/задачи за день»** = массив ID элементов 1218. Пусто ⇒ не заполнено |
| `ufCrm46_1742341577` | employee | Сотрудник |
| `ufCrm46_1742341767` | double | «По договорам (ч)» |
| `ufCrm46_1742341877` | iblock_element | **«Тип дня»** — числовой ID справочника iblock 27: 351 «Рабочий день», 352 «Отпуск», 355 «Отгул», 356 «Работа в выходной», 357 «Удалённая работа» (353/354+ недоступны без scope `lists`) |
| `ufCrm46_1745493416` | string | [тех] строковая денормализация «Типа дня» — только для аудита, фильтровать по числовому ID выше |

### entityTypeId **1218** — «Учёт рабочего времени» (categoryId=63, дочерний дню)
| Поле | Тип | Смысл |
|---|---|---|
| `parentId1208` | crm_entity | **ссылка на день** (1208.id) |
| `ufCrm48_1742996936` | crm | **Договор/ТКП/Шаблонные задачи** (для «Общие задачи подразделения» = `T512_2`) |
| `ufCrm48_1754894889` | string | [тех] ID договора (`2` для «Общих задач») |
| `ufCrm48_1744239302` | string | **Описание задачи** |
| `ufCrm48_1742996959` | double | **Количество часов** |

**Связь:** `1208.ufCrm46_1742997115` ⇄ `1218.parentId1208`.
Справочник договоров — СПА `1048` (17978) / `1052` (3622).

## Метамодель дополнение: CRM-дела дня (механизм «Выполнено»)

День уходит из фильтра «Запланированные» через завершение CRM-дела To-Do, привязанного к
карточке дня. `PROVIDER_ID` берётся из `config.activity_provider_id` (не хардкодить).

| Параметр | Значение |
|---|---|
| `PROVIDER_ID` | из `config.activity_provider_id` |
| `OWNER_TYPE_ID` | 1208 |
| `OWNER_ID` | id дня 1208 |
| Завершить | `crm.activity.update id=<id> fields:{COMPLETED:"Y"}` |

Создание учёта 1218 и закрытие дела — **два независимых шага**. Дело закрывать только после
успешной верификации учёта 1218.

## Рецепты

Прочитать дни за период (FR-1):
```bash
$PY "$CLIENT" crm.item.list --allow-unlisted --params \
 '{"entityTypeId":1208,"order":{"id":"desc"},
   "filter":{">=ufCrm46_1742342657":"2026-06-01","<=ufCrm46_1742342657":"2026-06-25"}}'
```

Прочитать учёты по дню:
```bash
$PY "$CLIENT" crm.item.list --allow-unlisted --params \
 '{"entityTypeId":1218,"filter":{"parentId1208":270608}}'
```

Отбор кандидатов для заполнения (часть 2): верхние 5 по `id desc` → дата в пределах
`edit_window_days=4` (календарные, включительно, по `ufCrm46_1742342657`) → «Тип дня»
(`ufCrm46_1742341877`) входит в whitelist `config.workday.day_type_work_ids` (по умолчанию
только `[351]` «Рабочий день» — отпуска/отгулы/выходные/удалёнка пропускаются) →
`ufCrm46_1742997115` пусто. Whitelist применяется и серверным фильтром `crm.item.list`
(`"@ufCrm46_1742341877":[351]`), и клиентской проверкой (защита от неизвестного/пустого типа);
фильтр действует только для `fill` — `export` (часть 1) выгружает дни всех типов без исключений.

Создать учёт (часть 2) — **двухфазно, с подтверждением**:
```bash
$PY "$CLIENT" crm.item.add --params \
 '{"entityTypeId":1218,"fields":{
     "parentId1208":<ID дня>,
     "ufCrm48_1742996936":"T512_2",
     "ufCrm48_1754894889":"2",
     "ufCrm48_1744239302":"Общие задачи подразделения",
     "ufCrm48_1742996959":8}}' --plan-only
# затем: --execute-plan <id> --confirm-write
```

Найти CRM-дела дня (FR-2.1.7, шаг перед закрытием):
```bash
$PY "$CLIENT" crm.activity.list --allow-unlisted --params \
 '{"filter":{"OWNER_TYPE_ID":1208,"OWNER_ID":<ID дня>,"PROVIDER_ID":"<из config>"}}'
```

Закрыть CRM-дело дня «Выполнено» (FR-2.1.7) — **только после верификации учёта**:
```bash
$PY "$CLIENT" crm.activity.update --params \
 '{"id":<ID дела>,"fields":{"COMPLETED":"Y"}}' --plan-only
# затем: --execute-plan <id> --confirm-write
```

## Политики и нюансы

- **Окно 4 дня (FR-2.1.3):** редактировать можно только дни `дата >= сегодня − 4` (Europe/Moscow). Жёстко проверять перед записью.
- **Идемпотентность:** перед `crm.item.add` перечитать день, убедиться, что `ufCrm46_1742997115` всё ещё пусто; после — проверить, что новый ID там появился.
- **Дефолты (дефолт+подтверждение):** описание = «Общие задачи подразделения», часы = 8.
- **Только чтение по умолчанию:** боевые записи в прод — лишь по явному разрешению.
- **«Выполнено» (FR-2.1.7) — ЗАКРЫТО (2026-06-26):** после создания 1218 обязателен второй шаг —
  `crm.activity.update COMPLETED=Y` на CRM-деле дня. Провайдер — из `config.activity_provider_id`.
  Дело закрывать только после успешной верификации учёта.
- Листинг типов (`crm.type.list`) недоступен (нет админ-прав) — entityTypeId брать из URL `/type/NNN/`.
