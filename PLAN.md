## Этап 4. Создание учёта (FR-2.1.6–2.1.10.4) [DONE — 2026-06-26]

**Цель:** заполнить отобранные дни.

1. **«Выполнено» (FR-2.1.7) — механизм УСТАНОВЛЕН (2026-06-26, исправление phase_4_01):**
   Ранее считалось, что `crm.item.add` дочернего 1218 достаточно. Боевой прогон опроверг это:
   день оставался в «Запланированных». Истинный механизм — **два независимых шага**:
   - Шаг 1: `crm.item.add entityTypeId=1218` с `parentId1208` = ID дня (авто-привязка OK, без доп. полей дня).
   - Шаг 2: `crm.activity.update id=<id дела> fields:{COMPLETED:"Y"}` — завершить CRM-дело.
     Дело ищется через `crm.activity.list filter:{OWNER_TYPE_ID:1208, OWNER_ID:<id дня>, PROVIDER_ID:<из config>}`.
     Провайдер — `config.activity_provider_id` (не хардкодить в логике).
     Шаг 2 выполняется **только после успешной верификации** шага 1.
   - Дополнительно: «ремонт»-проход — дни в окне 4 дней с непустым `ufCrm46_1742997115`
     (учёт есть, но дело открыто) → закрыть дела.

2. **Значения (FR-2.1.10.2/3, §6):** дефолт из `config.yaml` (описание «Общие задачи подразделения», часы 8) + подтверждение (`y`/новое значение/`skip`/`abort`; опция «применить ко всем»). `--no-interaction` → дефолты.

3. **Запись учёта:** `crm.item.add` `entityTypeId=1218`, `fields`:
   ```json

     "parentId1208": <ID дня>,
     "ufCrm48_1742996936": "T512_2",
     "ufCrm48_1754894889": "2",
     "ufCrm48_1744239302": "<описание>",
     "ufCrm48_1742996959": <часы>
   ```

   через `--plan-only` → подтверждение → `--execute-plan --confirm-write`.

4. **Верификация учёта:** перечитать день; убедиться, что `ufCrm46_1742997115` содержит новый ID.

5. **Закрытие дела (только при успешной верификации шага 4):** `crm.activity.update COMPLETED=Y`.
