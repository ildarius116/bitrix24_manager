## 0. Как рендерить для сверки (обязательная процедура верификации)

- **Qt-окно → PNG (offscreen):**
  `QT_QPA_PLATFORM=offscreen`, `PYTHONPATH=<корень>`, скрипт: `QApplication` → `MainWindow(cfg)` →
  `w.resize(1280,860)` → `w.show()` → `app.processEvents()` ×2 → `w.grab().save(out)`.
  (Шрифты в offscreen отсутствуют — кириллица будет квадратами; для проверки ВЁРСТКИ/цвета этого
  хватает. Для проверки текста — запускать реальное окно `gui_main.py` на машине заказчика.)
- **Макет HTML → PNG (эталон):**
  `chrome.exe --headless=old --disable-gpu --no-sandbox --user-data-dir=<tmp> --hide-scrollbars
  --screenshot=<out> --window-size=1280,880 --virtual-time-budget=25000 --run-all-compositor-stages-before-draw "file:///<путь>"`.
  ВНИМАНИЕ: при прошлом рендере внизу был `[bundle] error` и **область таблицы результатов не
  отрисовалась** — нужен полный эталон (см. «Открытые вопросы» §4).

## 2. ЭТАП B1 — Фильтр «Тип дня» (исправить отбор отпускных дней)

**Причина бага:** `src/workday.py::select_candidates` не смотрит тип дня → пустые дни «Отпуск» в
4-дневном окне попадают в кандидаты (жалоба заказчика; он в отпуске). Разведка завершена:

- Поле **«Тип дня» = `ufCrm46_1742341877`** (iblock_element, числовой ID; справочник iblock 27).
  Строковая копия `ufCrm46_1745493416` — только для аудита (фильтровать по числовому ID).
- Значения: **351 «Рабочий день»**, **352 «Отпуск»**, 355 «Отгул (весь день)»,
  356 «Работа в выходной/праздник», 357 «Удалённая работа» (353/354+ недоступны без scope `lists`).

**Изменения (src/ + config — после явного разрешения, это правка прод-логики):**
1. `config.yaml`: добавить в `fields:` → `workday_day_type: ufCrm46_1742341877`; и блок
   `workday: { day_type_work_ids: [351] }` (whitelist рабочих типов; **не хардкодить** коды в коде).
2. `src/config.py`: геттеры `field_workday_day_type` и `day_type_work_ids` (List[int]).
3. `src/workday.py`:
   - `read_days` / `select_candidates`: фильтровать по whitelist. Предпочтительно — **серверный
     фильтр в `crm.item.list`** (`filter: {"@ufCrm46_1742341877": [351]}` — подтверждено, что
     сервер фильтрует по этому полю), плюс защитная проверка на клиенте.
   - В `WorkdayDay` добавить поле `day_type_id` (читать из ответа), логировать пропуск:
     «День id=… пропущен: тип дня <id/подпись> не входит в рабочие (whitelist …)».
   - Применить тот же фильтр к `select_repair_days` (ремонт «Выполнено») — обсудить: ремонт
     закрывает уже заполненные дни; отпускной с учётом маловероятен, но whitelist безопаснее.
4. Тесты (`tester`): граничные — день 351 проходит; 352/355/неизвестный/пустой тип пропускаются;
   серверный фильтр замокать; обновить фикстуры `WorkdayDay` с `day_type_id`.

**Решение заказчика нужно:** whitelist только `[351]`, или включить `356`/`357`

## 3. ЭТАП B2 — Поведение датапикеров для `fill`

**Причина:** `fill` НЕ использует «Дата с/по» (они только для export); набор дней определяется
4-дневным окном + `limit=5`. При диапазоне 30.06–30.06 fill всё равно взял и 29.06 — поля ввели в
заблуждение, т.к. остаются активными в режиме редактирования.

**Варианты (выбрать с заказчиком):**
- **(A) Только UX, без src/:** в режиме «Редактирование» датапикеры **дизейблить/скрывать** и
  показать подпись «Заполнение работает по 4-дневному окну (top-5), даты не применяются».
  Минимально, безопасно.
- **(B) fill уважает выбранный день/диапазон (правка src/):** добавить в `run_fill`/отбор
  необязательный фильтр по диапазону как доп. ограничитель («заполнить только выбранное» — усиление
  гейта §5). Требует изменения `src/fill.py` сигнатуры и проводки из GUI.
- Реком.: начать с (A) сейчас; (B) — если заказчик хочет точечное заполнение по дате.
