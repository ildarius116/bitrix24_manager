# CLAUDE.md — Bitrix24 «Рабочий день»

## Обзор проекта

Автоматизация вкладки **«Рабочий день»** портала `bitrix.incomsystem.ru` через REST API.

Две части:
1. **Парсинг:** выгрузка таблицы «Рабочий день» в Excel за период (включая вложенные учёты).
2. **Редактирование:** автозаполнение «Работы/задачи за день» — создание записей «Учёта рабочего времени» через `crm.item.add`.

Стек: **Python 3.14** (в `venv/`), подход **REST-only** — никакого браузера/Playwright.
REST-клиент — из вендоренного скилла `.claude/skills/bitrix24-agent` (`scripts/bitrix24_client.py`).

## Ключевые правила работы

### 1. Фазность

Рабочий порядок: **ресёрч → план → код**. Реализационный код пишется только после явного указания пользователя. Агенты не расширяют код приложения проактивно.

### 2. Тимлид + делегирование

Оркестратор сам работу не выполняет — делегирует подчинённым агентам из `.claude/agents/`.

| Агент | Модель | Роль |
|---|---|---|
| `bitrix-analyst` | Opus | Анализ/ресёрч портала, read-only REST-пробы |
| `coder-simple` | Sonnet | Простые задачи коддинга |
| `coder-expert` | Opus | Сложный/многомодульный код, write-пайплайны |
| `tester` | Sonnet | pytest без сети |
| `reviewer` | Opus | Read-only код-ревью |
| `docs-keeper` | Sonnet | Документация и отчёты |

### 3. Трекинг задач

- Фазы/подфазы — в `tasks/` (имена `phase_<n>_<NN>_<slug>.md`; отложенные — `phase_delayed_n_NN_*`).
- По завершении — отчёт в `tasks_done/` (что и как сделано, DoD).
- **Исходные файлы в `tasks/` НЕ удалять** — хранятся вместе с отчётом.
- Индекс фаз — `tasks/INDEX.md`.

### 4. Безопасность

- Код вебхука — **СЕКРЕТ**: только в `.env` (вне git), никогда не печатать/не коммитить.
- В документации ссылаться обобщённо: «вебхук из `.env`», без конкретных кодов.
- После завершения работ — перегенерировать вебхук на портале (`/devops/` → «ПЕРЕГЕНЕРИРОВАТЬ»).
- **Аудит REST** — `.runtime/bitrix24_audit.jsonl` (путь переопределяется `B24_AUDIT_FILE` в `.env`).
  Пишутся только имена ключей (`param_keys`), метод, статус, duration, tenant — без значений и без
  кода вебхука. Файл в `.gitignore`; ротация вручную (аудит растёт без ограничений).
- **Журнал идемпотентности** — `.runtime/processed.json`; содержит только ID и временные метки,
  секретов нет. TTL задаётся `runtime.journal_ttl_days` в `config.yaml` (дефолт 7).

### 5. Политика записи в прод

По умолчанию — режим **«только чтение»** / `--dry-run`. Записи в продакшен — только по явному разрешению пользователя. Жёстко соблюдать **политику 4 дней**: редактировать можно только дни не старше 4 календарных дней (дата записи >= сегодня − 4, по `ufCrm46_1742342657`, таймзона Europe/Moscow).

### 6. Верификация GUI и честность отчётов (ЖЁСТКОЕ ПРАВИЛО)

> Внесено после провала Фазы 7: несколько итераций подряд GUI объявлялся «готовым/сверенным с
> макетом», хотя визуально ему НЕ соответствовал (строились не те контролы). Причина — подмена
> реальной проверки прокси (зелёные тесты, отчёт субагента, offscreen-рендер с нечитаемым текстом)
> и уверенные формулировки «готово/сверено» без фактической сверки. Так больше нельзя.

- **Источник истины по GUI — макет `design/bitrix24-console-standalone.html`** (интерактивный SPA),
  а НЕ текстовый спек. Его нужно ОТКРЫВАТЬ/рендерить и смотреть — в том числе переключая режимы
  (Парсинг/Редактирование) и методы (По-умолчанию/Индивидуально). Набор контролов зависит от
  состояния; сверять КАЖДОЕ состояние, а не только стартовый экран.
- **Сверять глазами порежимными PNG-рендерами:** и макет, и собранное окно — в каждом режиме
  (`gui_parse.png`, `gui_edit.png`, `gui_indiv.png` и т.п.). Строки из бандла макета можно
  вытащить grep'ом русских строк (минифицированный HTML) — так виден точный состав контролов.
- **Что НЕ является доказательством соответствия:** зелёный `pytest`, «окно инстанцируется без
  исключений», отчёт субагента, offscreen-рендер (кириллица = квадраты — годится для вёрстки/цвета,
  НЕ для текста). Финальную проверку текста делает пользователь на реальной машине (`gui_main.py`).
- **Не отчитываться «готово/сверено/соответствует» без фактической визуальной сверки.** Никогда не
  утверждать, что проверка выполнена, если она не выполнялась. Формулировать так: что именно
  сделано и что НЕ проверено — пусть пользователь судит.
- **Честно разделять «нарисовано в UI» и «реально работает».** Если контрол présent визуально, но
  не проведён в логику (например, чекбоксы дней/поля не влияют на запись) — это ЯВНО указывать в
  отчёте и в доках («Известные ограничения»), а не выдавать за готовую функциональность.

## Метамодель (кратко)

Подробности — в `PRD.md §2.6` и в скилле `bitrix24-workday`.

- `entityTypeId=1208` — «Рабочий день» (categoryId=61, одна запись/день).
- `entityTypeId=1218` — «Учёт рабочего времени» (categoryId=63, дочерний к дню).
- Связь: `1208.ufCrm46_1742997115` (массив ID) ⇄ `1218.parentId1208`.
- Конкретные коды полей (`ufCrm46_*`, `ufCrm48_*`) — в `config.yaml` и PRD §2.6; в логике **не хардкодить**, брать из config.

## Структура проекта

```
bitrix24/
├── PRD.md                   # Требования (включая метамодель §2.6)
├── PLAN.md                  # План реализации (этапы, REST-вызовы)
├── CLAUDE.md                # Этот файл
├── config.yaml              # entityTypeId, коды полей, дефолты, окно 4 дней
├── .env                     # СЕКРЕТ: код вебхука (в .gitignore)
├── .env.example             # Шаблон без секретов
├── requirements.txt
├── main.py                  # CLI: подкоманды export / fill
├── src/                     # Модули приложения (пишутся по указанию)
├── scripts_discovery.py     # Read-only утилита ресёрча (артефакты в out/)
├── out/                     # Excel-выгрузки, логи, артефакты (вне git)
├── tasks/                   # Спецификации фаз/подфаз
├── tasks_done/              # Отчёты о завершённых задачах
└── .claude/                 # Скиллы и агенты (локально, в .gitignore)
    ├── skills/
    │   ├── bitrix24-agent/      # REST-клиент (транспорт, безопасность)
    │   └── bitrix24-workday/    # Метамодель + рецепты (надстройка над agent)
    └── agents/                  # Подчинённые агенты
```

> `.claude/` находится в `.gitignore` — скиллы и агенты хранятся **локально**, не в git.

## Команды

Установка зависимостей:
```bash
venv/Scripts/python.exe -m pip install -r requirements.txt
```

Smoke-доступ (read-only, проверка вебхука из `.env`):
```bash
venv/Scripts/python.exe .claude/skills/bitrix24-agent/scripts/bitrix24_client.py \
  user.current --params '{}' --allow-unlisted
```

Тесты (без сети):
```bash
venv/Scripts/python.exe -m pytest -q
```

## Скиллы

При любой работе с Bitrix24 в этом проекте использовать скилл **`bitrix24-workday`** (метамодель + готовые рецепты). Он надстройка над общим **`bitrix24-agent`** (транспорт, allowlist, plan→execute, идемпотентность, аудит, ретраи, rate-limit).

Справочник доступных REST-методов: `.claude/skills/bitrix24-workday/references/available-methods.md`.
