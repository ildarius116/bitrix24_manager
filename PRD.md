# PRD — Автоматизация вкладки «Рабочий день» в Bitrix24

**Документ:** Product Requirements Document
**Проект:** `bitrix24`
**Дата составления:** 2026-06-25
**Статус:** v2.0 — REST-only (доступ и модель данных подтверждены ресёрчем, см. §2.6). Реализация — по `PLAN.md`, код по отдельному указанию.
**Портал:** https://your-portal.bitrix24.ru (self-hosted Bitrix24, on-premise)
**Целевая страница:** https://your-portal.bitrix24.ru/page/timemenedzher/rabochiy_den/?login=yes

---

## 1. Обзор и цель

Проект автоматизирует рутинную работу с кастомной вкладкой **«Рабочий день»** портала Bitrix24 компании Example.

Две группы задач:

1. **Парсинг (чтение/выгрузка):** выгрузка таблицы «Рабочий день» в Excel за заданный диапазон дат, включая рекурсивный сбор содержимого вложенных окон столбца «Работы/задачи за день», плюс отдельный лист с группировкой по «Описанию задачи».
2. **Редактирование (запись):** автозаполнение столбца «Работы/задачи за день» для последних незаполненных записей в пределах разрешённого политикой 4-дневного окна — через цепочку модальных окон («Заполнить работы по договорам» → «Выполнено» → «Учёт рабочего времени» → «+ Новый элемент» → заполнение и «СОХРАНИТЬ»).

Конечный результат — CLI-инструмент на Python, который запускается оператором (или по расписанию) и выполняет обе группы задач.

## 2. Результаты ресёрча

### 2.1. Тип портала и сущности

- Портал **self-hosted** (`your-portal.bitrix24.ru`), не облачный `*.bitrix24.ru`.
- Страница `/page/timemenedzher/rabochiy_den/` — **кастомная страница** портала (контентный раздел с встроенным компонентом/гридом), а не штатный модуль «Рабочий день» (Тайм-менеджер) из коробки.
- Сущности из задачи — «Учёт рабочего времени», «Договор/ТКП/Шаблонные задачи», «Общие задачи подразделения», «Заполнить работы по договорам» — с высокой вероятностью являются **кастомными смарт-процессами (СПА / CRM dynamic types)**. В REST они адресуются через универсальный API `crm.item.*` с индивидуальным `entityTypeId` (≥ 1000 для пользовательских типов). Эти ID **неизвестны** и требуют выяснения на портале.

### 2.2. Установленный скилл `bitrix24-agent`

- Вендорен **в проект**: `.claude/skills/bitrix24-agent` (источник: https://github.com/vrtalex/bitrix24-skill, MIT). Версионируется вместе с репозиторием.
- Назначение: **только REST API** Bitrix24 (webhook / OAuth 2.0). Ключевые методы: `crm.item.list/get/add/update`, `crm.item.fields`, `tasks.task.*`, `timeman.status/open/close`, `batch`, `user.current`.
- Содержит безопасный CLI-клиент `scripts/bitrix24_client.py` (allowlist, plan→execute, идемпотентность, аудит, ретраи, rate-limit).
- **Ограничение:** скилл НЕ умеет управлять браузером (клики по кнопкам, модальные окна, гиперссылки). Все шаги части 2 описаны как UI-действия, для которых у скилла нет прямого аналога без знания REST-эквивалентов смарт-процессов.

### 2.3. Решение по доступу (ОБНОВЛЕНО — найден REST-вебхук)

- На портале есть конструктор **входящих вебхуков** (`/devops/edit/...`), позволяющий создавать REST-вебхуки с произвольным набором scope. Полный перечень доступных методов сохранён в `generator.txt` (включает `crm.item.*`, `crm.type.*`, `timeman.*`, `task.elapseditem.*`, `tasks.task.*`).
- Это **разблокирует REST/гибрид**: «чистый REST» теперь возможен. Браузерная автоматизация остаётся как **fallback** (логин усложнён 2FA — см. ниже).
- **Найденный тестовый вебхук** (user 1, scope `crm`) пригоден для дискавери, но имеет ограничения по правам (см. §2.4). Это **секрет**: после работ его следует перегенерировать.
- **2FA на входе: ДА** — это дополнительный аргумент в пользу REST (вебхук обходит 2FA), т.к. браузерный логин потребует ручного ввода кода.

Только после этого можно прочитать коды полей обоих процессов и определить природу кнопки «Выполнено».

### 2.5. Прочие уточнения заказчика

- «Последние 5 записей» (FR-2.1.1) = **верхние 5 строк грида** (= сортировка `1208` по `id desc`).
- 2FA на входе = ДА (аргумент за REST).
- Значения «Описание задачи»/«Количество часов» = дефолт + подтверждение.

**Связь:** `1208.ufCrm46_1742997115` (массив) ⇄ `1218.parentId1208`. Подтверждено: день 269698 (24.06) → учёт 162468; день 270608 (25.06) → учёт 163025.

**Эталон для создания** записи «Общие задачи подразделения» (копируем значения существующего 1218): `parentId1208=<id дня>`, `ufCrm48_1742996936='T512_2'`, `ufCrm48_1754894889='2'`, `ufCrm48_1744239302=<описание>`, `ufCrm48_1742996959=<часы>`.

**Механизм «Выполнено» (FR-2.1.7) — ПОДТВЕРЖДЁН (2026-06-26):** день уходит из фильтра
«Запланированные» **не** через поля дня 1208, а через завершение CRM-дела типа To-Do,
привязанного к карточке дня. Создание учёта 1218 и закрытие дела — **два независимых шага**:

1. `crm.item.add entityTypeId=1218 fields:{parentId1208, ...}` — создать учёт (авто-привязка OK).
2. `crm.activity.update id=<id дела> fields:{COMPLETED:"Y"}` — завершить CRM-дело.
   Дело ищется через `crm.activity.list filter:{OWNER_TYPE_ID:1208, OWNER_ID:<id дня>, PROVIDER_ID:<из config>}`.
   Провайдер (`activity_provider_id`) хранится в `config.yaml`, не хардкодится.
   Шаг 2 выполняется **только** после успешной верификации шага 1.

## 3. Архитектурное решение

> **Вердикт (ФИНАЛ):** Проект **REST-only**. Доступ есть (вебхук user 1 = Владелец записей, scope `crm,task,timeman,user`), модель данных подтверждена (§2.6). Обе части задачи реализуются через `crm.item.*`:
> - **Часть 1 (парсинг):** `crm.item.list`/`crm.item.get` по `entityTypeId=1208` (+ дочерние 1218) → Excel.
> - **Часть 2 (редактирование):** `crm.item.add` элемента `1218` с `parentId1208` = ID дня → привязка к «Рабочему дню».
>
> Браузерная автоматизация исключена. Единственный открытый момент — семантика кнопки «Выполнено» (см. §2.6), решается на этапе реализации в режиме «только чтение/сначала наблюдение».

### 3.1. Почему REST (итог)

| Критерий | REST API | Браузер (исключён) |
|---|---|---|
| Доступ | ✅ вебхук есть (scope `crm,task,timeman,user`) | требует логин + обход 2FA |
| 2FA | ✅ не мешает | ⛔ блокирует автоматизацию |
| Знание модели | ✅ entityTypeId 1208/1218 и поля подтверждены | — |
| Устойчивость | ✅ стабилен | ⚠️ ломается при смене вёрстки |
| Скорость | ✅ headless, быстро | ⚠️ медленно |

### 3.2. Архитектура (REST-only)

## 4. Технологический стек

> **Решение:** проект **REST-only**. Браузерная автоматизация (Playwright) исключена — вся задача (части 1 и 2) покрывается REST `crm.item.*`. Это подтверждено дискавери (§2.6).

- **Язык:** Python 3.14 (в проекте уже установлен в `venv/`).
- **REST-клиент:** установленный скилл **`bitrix24-agent`** (`scripts/bitrix24_client.py`, stdlib, без зависимостей; allowlist/packs, plan→execute, идемпотентность, аудит, ретраи, rate-limit).
- **Excel:** `openpyxl` (несколько листов), при необходимости `pandas` для группировки (часть 1).
- **Конфигурация:** `python-dotenv` (`.env`, секрет вебхука) + `config.yaml` (несекретные параметры).
- **Хранение секретов:** код вебхука только в `.env` (вне git).
- **Логирование:** стандартный `logging` + структурный лог запусков, маскирование секретов.
- **CLI:** `argparse`.
- **Исключено:** Playwright/Selenium, браузерный логин, обход 2FA (не нужен — REST использует вебхук).

## 5. Функциональные требования

### FR-0. Доступ (REST)

- FR-0.1. Аутентификация через **входящий вебхук** (`B24_DOMAIN`, `B24_WEBHOOK_USER_ID`, `B24_WEBHOOK_CODE` из `.env`), scope `crm,task,timeman,user`. 2FA не задействуется.
- FR-0.2. Smoke-проверка доступа при старте (`user.current`); понятная ошибка при `insufficient_scope`/`INVALID_CREDENTIALS`.
- FR-0.3. Код вебхука — секрет: только в `.env`, маскировать в логах.

### FR-1. Парсинг (часть 1)

**FR-1.1. Выгрузка таблицы в Excel за диапазон дат.**
- FR-1.1.1. Параметры запуска: `--date-from` и `--date-to` (формат `дд.мм.гггг` или ISO).
- FR-1.1.2. Применить фильтр по датам на странице (или отфильтровать локально после выгрузки).
- FR-1.1.3. Считать все строки таблицы со всеми видимыми столбцами (включая «Название», «Работы/задачи за день» и пр.).
- FR-1.1.4. **Рекурсия по столбцу «Работы/задачи за день»:** в ячейке несколько вложенных окон; необходимо открыть и извлечь **содержимое окна «Информация о проделанной работе»** для каждой строки. Сохранять как отдельный (под)набор данных, связанный со строкой.
- FR-1.1.5. Записать результат на **основной лист** Excel (одна строка таблицы = одна строка листа; содержимое вложенных окон — в отдельные колонки или объединённую ячейку с разделителями — формат уточняется, см. §12).
- FR-1.1.6. **Определение временных рамок данных.** При парсинге дополнительно вычислить фактический диапазон дат в выгруженных данных (минимальная и максимальная дата записей, число дней/записей). Это нужно, чтобы:
  - подтвердить, что весь заданный период `--date-from..--date-to` действительно покрыт данными (нет «дыр»/обрезки пагинацией);
  - зафиксировать реальные границы в метаданных выгрузки (лог + отдельная мета-строка/лист в Excel).
  - Если фактические рамки уже данных уже заданного периода — предупредить оператора (часть записей могла не подгрузиться).

**FR-1.2. Группировка на дополнительном листе.**
- FR-1.2.1. На **втором листе** Excel сгруппировать содержимое столбца «Работы/задачи за день» по значению окна **«Описание задачи»**.
- FR-1.2.2. Для каждой группы (уникальное «Описание задачи») — агрегировать связанные записи (перечень, суммарные часы — если применимо, кол-во вхождений).

### FR-2. Редактирование (часть 2): автозаполнение «Работы/задачи за день»

Алгоритм (точно по ТЗ заказчика):

- **FR-2.1.1.** Найти **последние 5 записей** вкладки «Рабочий день».
- **FR-2.1.2.** Прочитать столбец **«Название»**: содержимое — гиперссылка, в тексте которой есть дата в формате `дд.мм.гггг`. Извлечь дату из текста ссылки.
- **FR-2.1.3.** **Фильтр по 4-дневному окну:** оставить только записи, у которых `(текущая_дата − дата_записи) ≤ 4 дня`. Записи старше — **редактировать запрещено политикой**, пропускаются.
  > Уточнить семантику «4 дней»: календарные или рабочие дни; включительно/исключительно. По умолчанию — **календарные, включительно** (`дата_записи ≥ сегодня − 4 дня`). Сегодня = `2026-06-25` → редактируемы записи от `2026-06-21` и новее.
- **FR-2.1.4.** Для оставшихся записей проверить столбец **«Работы/задачи за день»** на заполненность.
- **FR-2.1.5.** Оставить только записи, где «Работы/задачи за день» **пусто**.
- **FR-2.1.5a.** **Фильтр по «Типу дня» (`ufCrm46_1742341877`, §2.6).** Оставить только записи,
  чей числовой «Тип дня» входит в whitelist `config.yaml → workday.day_type_work_ids` (по
  умолчанию — только `[351]` «Рабочий день», решение заказчика). Дни «Отпуск» (352), «Отгул» (355),
  «Работа в выходной» (356), «Удалённая работа» (357), а также дни с пустым/неизвестным типом —
  **пропускаются** (не заполняются), с логом причины. Применяется серверный фильтр `crm.item.list`
  по `@ufCrm46_1742341877` + защитная клиентская проверка на тот же whitelist; тот же фильтр

## 6. Сбор значений «Описание задачи» и «Количество часов» (решение заказчика: «Дефолт + подтверждение»)

- Источник значений по умолчанию — `config.yaml`:
  - `defaults.task_description` (например, `"Общие задачи подразделения"`);
  - `defaults.hours` (например, `8`).
- Перед нажатием «СОХРАНИТЬ» по каждой записи (или один раз на пакет — уточнить, см. §12) скрипт **выводит предлагаемые значения** и просит подтверждение:
  - Enter / `y` — принять дефолт;
  - ввод нового значения — переопределить для текущей записи;
  - `skip` — пропустить запись;
  - `abort` — прервать выполнение.
- Режим `--no-interaction` (для автозапуска) использует дефолты без подтверждения.
- Все применённые значения логируются.

## 7. Нефункциональные требования

- **NFR-1. Безопасность секрета.** Код вебхука — только в `.env` (вне git). Маскирование в логах. Никаких секретов в коде/репозитории.
- **NFR-2. Идемпотентность записи.** Перед `crm.item.add` повторно перечитать день и убедиться, что `ufCrm46_1742997115` всё ещё пусто (защита от двойного запуска). Вести локальный журнал обработанных дней (id/дата) с TTL.
- **NFR-3. Безопасный режим (dry-run).** Флаг `--dry-run`: вывести готовый payload `crm.item.add`, но не выполнять запись (использовать `--plan-only` клиента скилла).
- **NFR-4. Надёжность REST.** Ретраи с backoff на `QUERY_LIMIT_EXCEEDED`/5xx (есть в клиенте), таймауты, проверка ответа после записи (перечитывание).
- **NFR-5. Соблюдение политик портала.** Жёсткое соблюдение 4-дневного окна (FR-2.1.3) по полю даты `ufCrm46_1742342657`. Никаких записей вне окна.
- **NFR-6. Логирование и аудит.** Структурный лог: что прочитано, что отфильтровано и почему, какой payload записан. Аудит REST-вызовов (`B24_AUDIT_FILE`).
- **NFR-7. Конфигурируемость метамодели.** `entityTypeId` и коды полей (`ufCrm46_*`, `ufCrm48_*`) вынести в `config.yaml`, чтобы пережить изменения портала без правки логики.
- **NFR-8. Производительность.** Батч-запросы (`batch`) для чтения дней+учётов; уважать rate-limit (2/с).
