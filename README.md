# Bitrix24 — автоматизация вкладки «Рабочий день»

Парсинг и автозаполнение вкладки «Рабочий день» портала `your-portal.bitrix24.ru`
через **REST API** (смарт-процессы `crm.item.*`). Постановка — в [PRD.md](PRD.md),
план реализации — в [PLAN.md](PLAN.md).

## Команды

```bash
# Выгрузка «Рабочего дня» за период в Excel (read-only)
venv/Scripts/python.exe main.py export --date-from 01.06.2026 --date-to 25.06.2026

# Автозаполнение учётов — по умолчанию DRY-RUN (ничего не пишет, показывает план)
venv/Scripts/python.exe main.py fill --dry-run

# Боевая запись в прод — ТОЛЬКО с явным гейтом --confirm-write (без --dry-run).
# Соблюдается политика 4 дней; перед каждой записью день перечитывается (окно + пустота).
# Боевой режим: crm.item.add 1218 → верификация → закрытие CRM-дела «Выполнено»
# (crm.activity.update COMPLETED=Y, FR-2.1.7) + «ремонт» открытых дел у заполненных дней.
# Повторный запуск не плодит дублей: журнал .runtime/processed.json (TTL) + перечитывание дня.
venv/Scripts/python.exe main.py fill --confirm-write
#   --no-interaction — без вопросов, дефолты из config.yaml (описание/часы)

# Тесты (без сети)
venv/Scripts/python.exe -m pytest -q
```

> Без `--confirm-write` (или вместе с `--dry-run`) команда `fill` всегда работает в режиме
> dry-run и в прод ничего не пишет.

## Подготовка

```bash
venv/Scripts/python.exe -m pip install -r requirements.txt
cp .env.example .env      # впиши B24_WEBHOOK_CODE (секрет!)
```

REST-клиент берётся из вендоренного в проект скилла `.claude/skills/bitrix24-agent`.

## Безопасность

Код вебхука — секрет, только в `.env` (вне git). После завершения работ
**перегенерируй вебхук** на портале (`/devops/` → «ПЕРЕГЕНЕРИРОВАТЬ»).

Аудит REST (`.runtime/bitrix24_audit.jsonl`) и журнал идемпотентности (`.runtime/processed.json`)
секретов не содержат — пишутся только имена параметров, ID и временные метки. Каталог `.runtime/`
в `.gitignore`; аудит растёт без ротации (чистить вручную).
