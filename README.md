## Команды

```bash
# Выгрузка «Рабочего дня» за период в Excel (read-only)
venv/Scripts/python.exe main.py export --date-from 01.06.2026 --date-to 25.06.2026

# Автозаполнение учётов — по умолчанию DRY-RUN (ничего не пишет, показывает план)
venv/Scripts/python.exe main.py fill --dry-run

# Боевая запись в прод — ТОЛЬКО с явным гейтом --confirm-write (без --dry-run).
# Соблюдается политика 4 дней; перед каждой записью день перечитывается (окно + пустота).
# Боевой режим: crm.item.add 1218 → верификация → закрытие CRM-дела «Выполнено»
# (crm.activity.update COMPLETED=Y, FR-2.1.7) + «ремонт» открытых дел у заполненных дней.
# Повторный запуск не плодит дублей: журнал .runtime/processed.json (TTL) + перечитывание дня.
venv/Scripts/python.exe main.py fill --confirm-write
#   --no-interaction — без вопросов, дефолты из config.yaml (описание/часы)

# Тесты (без сети)
venv/Scripts/python.exe -m pytest -q
```

> Без `--confirm-write` (или вместе с `--dry-run`) команда `fill` всегда работает в режиме
> dry-run и в прод ничего не пишет.

## Как это работает

Утилита запускается через `main.py` с одной из двух подкоманд. При любом запуске сначала
выполняется общий старт: загрузка `config.yaml` + секрет вебхука из `.env`, логирование с
маскированием секрета и read-only smoke-проверка доступа (`user.current`). Нет доступа — выход
с кодом 2, никаких действий.

**`export`** (только чтение) — выгрузка «Рабочего дня» за период в Excel:
дни `1208` за `[--date-from; --date-to]` → связанные учёты `1218` → ФИО сотрудников →
файл `out/workday_<от>_<до>.xlsx` (3 листа: Данные / Группировка / Метаданные).

**`fill`** — автозаполнение «Работ/задач за день» в окне 4 дней (день не старше сегодня−4, МСК):

1. **Отбор** — до 5 последних *пустых* дней в окне, дни с «Типом дня» вне whitelist
   (`config.yaml → workday.day_type_work_ids`, по умолчанию только `351` «Рабочий день»)
   **исключаются**: отпуск, отгул, работа в выходной, удалённая работа — не заполняются
   (см. PRD §2.6, FR-2.1.5a). Фильтр применяется и на сервере (`crm.item.list`), и повторно на
   клиенте — как защита от неизвестного/пустого типа дня.
2. **Значения** — описание + часы: в интерактиве спрашивает, иначе берёт дефолты из `config.yaml`.
3. **Создание** — перед записью перечитывает день (окно + пустота ещё раз), затем `crm.item.add` (`1218`).
4. **Верификация** — перечитывает: связь день↔учёт, описание, часы.
5. **Закрытие дела «Выполнено»** (FR-2.1.7) — только если верификация прошла:
   `crm.activity.update COMPLETED=Y`, день уходит из «Запланированных».
6. **«Ремонт»** — отдельный проход по дням, где учёт уже есть, но дело осталось открытым — дозакрывает дело.
7. **Сводка** — таблица статусов в логе.

Защита от дублей: журнал `.runtime/processed.json` (TTL из `runtime.journal_ttl_days`) +
перечитывание дня перед каждой записью. **Без `--confirm-write` запись физически не выполняется.**

### Порядок работы

```bash
# 1. Выгрузка в Excel — безопасно, только чтение
venv/Scripts/python.exe main.py export --date-from 01.06.2026 --date-to 25.06.2026

# 2. Предпросмотр заполнения — НИЧЕГО не пишет (dry-run по умолчанию)
venv/Scripts/python.exe main.py fill --dry-run

# 3. Боевая запись — только с явным гейтом
venv/Scripts/python.exe main.py fill --confirm-write                  # с вопросами по каждому дню
venv/Scripts/python.exe main.py fill --confirm-write --no-interaction # без вопросов, дефолты из config.yaml
```

### Что вводить в интерактиве (`fill` с вопросами)

По каждому дню показываются предложенные описание/часы, ввод:

| Ввод | Действие |
|------|----------|
| `Enter` | принять предложенное значение |
| текст / число | переопределить описание / часы |
| `skip` | пропустить этот день |
| `abort` | прервать весь пакет |
| `all` | применить дефолт ко всем оставшимся дням |

После первого дня утилита спросит «применить эти значения ко всем оставшимся? [y/N]».

### Настройка значений по умолчанию

Описание, часы и прочие дефолты — в `config.yaml` (`defaults.task_description`, `defaults.hours`,
`edit_window_days`, `runtime.journal_ttl_days`). Коды полей и `entityTypeId` там же — в логике не
хардкодятся. Отдельно: `fields.workday_day_type` — код поля «Тип дня» (`ufCrm46_1742341877`) и
`workday.day_type_work_ids` — whitelist числовых ID «рабочих» типов дня для `fill` (по умолчанию
`[351]`, см. «Как это работает» выше и PRD §2.6).

## Графический интерфейс (GUI)

Десктопная GUI-обёртка на **PySide6** над теми же командами `export`/`fill` — для тех, кто
предпочитает окно консоли. Бизнес-логика не дублируется: GUI вызывает те же модули `src/`, что и
`main.py`; CLI и GUI работают параллельно и не конфликтуют.

Запуск:

```bash
venv\Scripts\python.exe gui_main.py
```

При старте, как и в CLI, выполняется загрузка `config.yaml` + секрета вебхука из `.env`,
настройка логирования с маскированием секрета, затем открывается окно и применяется тема.

> Раздел описывает **фактическое** состояние GUI по коду (`gui/control_panel.py`,
> `gui/main_window.py`, `gui/console_panel.py`, `gui/settings_dialog.py`, `gui/title_bar.py`),
> сверенному покадровыми PNG-рендерами обоих режимов. Несколько предыдущих версий этого раздела
> описывали контролы, которых в коде не было (см. «Известные ограничения GUI» ниже и
> ретроспективу в `tasks_done/phase_7_done.md`).

Компоновка окна (сверху вниз):

1. Системная рамка Windows (окно не безрамочное).
2. **Синяя титульная полоса-баннер** (`gui/title_bar.py`) — «Bitrix24 · Рабочий день», НАД меню.
3. **Меню** «Файл · Настройки · Вид · Справка», в правом углу меню-бара — переключатель темы
   (🌙/☀) и **индикатор вебхука** «● Проверка…» / «● Вебхук подключён» / «● Вебхук недоступен»
   (обновляется по результату smoke-проверки `user.current`, без кода вебхука).
   «Настройки → Значения по-умолчанию…» открывает `SettingsDialog` (`gui/settings_dialog.py`) —
   модальное окно с полями «Описание задачи»/«Часы» (пре-филл из `config.yaml → defaults`),
   которое редактирует **те же** значения, что и инлайн-поля «Описание задачи по-умолчанию»/
   «Часов по-умолчанию» в левой колонке (см. ниже) — это не единственное место их редактирования,
   а зеркало.
4. Двухколоночная область (`QSplitter`): слева `ControlPanel`, справа тёмная `ConsolePanel`.
5. **Статусбар** — слева «Готов · Режим: Парсинг/Редактирование», справа
   «`<домен>` · REST · 1208/1218» (без кода вебхука).

### Левая колонка (`gui/control_panel.py`) — режим-зависимая

Сверху — взаимоисключающий переключатель режима **«⬇ Парсинг»** (export) /
**«✎ Редактирование»** (fill). Набор контролов ниже зависит от выбранного режима; общая для обоих
режимов таблица результатов (`ResultTable`) показывается только когда в ней есть строки (иначе
скрыта, а её место занимает растягивающийся спейсер, чтобы синяя CTA была прижата к низу колонки).

**Режим «Парсинг»:**

- «Начальная дата» / «Конечная дата» (`QDateEdit`, `dd.MM.yyyy`) + live-подпись «Период: …».
- CTA внизу — **«⬇ Получить выписку»**.

**Режим «Редактирование»** (как в макете `design/bitrix24-console-standalone.html`):

- **«Метод заполнения задач»** — выпадающий список (`QComboBox`): **«По-умолчанию»** (`auto`,
  выбран по умолчанию) / **«Индивидуально»** (`interactive`).
- **«Описание задачи по-умолчанию»** (`QLineEdit`) и **«Часов по-умолчанию»**
  (`QDoubleSpinBox` + подпись «ч») — значения по умолчанию для заполнения; те же значения, что
  редактируются в диалоге «Настройки → Значения по-умолчанию».
- **«ДНИ ДЛЯ ЗАПОЛНЕНИЯ»** — 5 строк с чекбоксами: «Сегодня», «1 день назад» … «4 дня назад»
  (4-дневное окно) с датами; по умолчанию отмечена только «Сегодня». Вид строки зависит от метода:
  - метод «По-умолчанию» + день отмечен → статичный текст «`<описание>` · `<часы>` ч»;
  - метод «Индивидуально» + день отмечен → у строки появляются **поля ввода** (описание + часы)
    для индивидуального значения этого дня.
- **«Запланировать»** — чекбокс + `QTimeEdit` (по умолчанию 09:00) + подпись «24ч».
- CTA внизу — **«Выполнить»**.

### Известные ограничения GUI (честно, без приукрашивания)

1. **Чекбоксы «ДНИ ДЛЯ ЗАПОЛНЕНИЯ» и инлайн-поля описания/часов по дням (в «Индивидуально») —
   пока ТОЛЬКО UI-элементы.** Они не управляют реальной записью: какие дни фактически
   обрабатываются, по-прежнему решает логика `fill` (4-дневное окно + top-5, `src/fill.py`), а в
   методе «Индивидуально» описание/часы по-прежнему запрашиваются через per-day
   `gui/confirm_dialog.py` во время прогона — значения, введённые в строках дней колонки, в запись
   не передаются. Чтобы это заработало по-настоящему, нужно: (а) прокинуть значения строк в
   воркер (сторона GUI, без правки `src/`) и (б) ограничить фактическую запись отмеченными днями
   (требует правки `src/fill.py`). Ни то, ни другое не сделано.
2. **«Запланировать» — визуальный элемент.** Планирование запуска по времени не реализовано;
   нажатие «Выполнить» всегда запускает операцию немедленно, независимо от чекбокса/времени.
3. **Серверный фильтр «Тип дня» (`[351]`, §2.6) не проверялся вживую на портале** — покрыт только
   синтетическими тестами; клиентская защитная проверка (`_day_type_allowed`) подстраховывает на
   случай, если сервер фильтр проигнорирует.
4. **Offscreen-рендеры для визуальной сверки показывают кириллицу квадратами** (нет системных
   шрифтов в offscreen-окружении CI/агента) — компоновку и расположение блоков так проверить можно,
   а точный текст/переносы — только запуском `gui_main.py` на реальной машине.

### Правая тёмная консоль (`gui/console_panel.py`)

Всегда тёмная (`#1e1e1e`), независимо от темы приложения:

- **ПРОГРЕСС** — заголовок, `%`, `QProgressBar` (indeterminate во время операции), строка
  состояния («Ожидание»/«Выполняется…»).
- **ЖУРНАЛ** — заголовок, вкладки-фильтры **[Все] [REST] [OK] [Ошибки]**, счётчик «N стр.»,
  кнопки экспорта журнала в файл (⤓) и очистки (🗑). Использует `QLoggingHandler` и маскирование
  секрета (`SecretMaskingFilter`); REST-записи подсвечиваются отдельным цветом. Лог —
  моноширинный `QTextEdit` (read-only).

### Таблица результатов и диалог подтверждения

- **Таблица результатов (`gui/result_table.py`)** — строки `export`/`fill` с цветовой индикацией
  статусов (`filled`/`repaired`/`skipped`/`already-closed`/`error`/`aborted`); служебные
  `raw`-поля Bitrix24 не выводятся.
- **dry-run в GUI нет.** Заполнение всегда запускается как боевое (`FillWorker(..., dry_run=False,
  ...)`), но **перед любой записью** обязателен диалог подтверждения:
  - `_confirm_live_write` (`gui/main_window.py`) — `QMessageBox` «Боевой режим — подтверждение
    записи» с кнопками «Подтвердить запись» / «Отмена» (дефолт — «Отмена»); показывается
    безусловно для **обоих** методов, до старта воркера. Причина безусловности: запись идёт не
    только при создании учётов, но и в «ремонте» (закрытие дел «Выполнено»,
    `crm.activity.update`), который не проходит через per-day диалог.
  - Метод **«Индивидуально»** — ДОПОЛНИТЕЛЬНО по каждому дню показывается модальный
    `gui/confirm_dialog.py` с предложенными описанием/часами и кнопками «Пропустить» /
    «Применить ко всем» / «Отмена» / «OK»; закрытие крестиком/Esc трактуется как «Отмена».
  Предпросмотр без записи остаётся доступен только в CLI: `main.py fill --dry-run`.

Тема и стили (`gui/theme.py`, `resources/styles/main.qss`) вендорятся из дизайн-системы
`United_Stand_Platform` (USP) — без живой зависимости на сам проект USP.

Политика 4 дней, фильтр «Тип дня» и идемпотентность (журнал `.runtime/processed.json`) работают
так же, как в CLI, — эта логика не дублируется и не переопределяется в GUI-коде (см. CLAUDE.md
§4/§5).

## Безопасность

Код вебхука — секрет, только в `.env` (вне git). После завершения работ
**перегенерируй вебхук** на портале (`/devops/` → «ПЕРЕГЕНЕРИРОВАТЬ»).

Аудит REST (`.runtime/bitrix24_audit.jsonl`) и журнал идемпотентности (`.runtime/processed.json`)
секретов не содержат — пишутся только имена параметров, ID и временные метки. Каталог `.runtime/`
в `.gitignore`; аудит растёт без ротации (чистить вручную).
