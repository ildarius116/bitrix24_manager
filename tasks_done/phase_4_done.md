## Что сделано по подзадачам

### 4_01 — Механизм кнопки «Выполнено» (read-only дискавери)

Вывод (подтверждён сравнением пустого/заполненного дня и `crm.item.fields`): кнопка «Выполнено» —
UI-обёртка над созданием дочернего «Учёта рабочего времени» (1218) с привязкой через
`parentId1208`. **Двусторонняя связь создаётся ядром Bitrix автоматически: достаточно одного
`crm.item.add` 1218** — портал сам добавляет ID учёта в `1208.ufCrm46_1742997115` и пересчитывает
агрегаты дня. У 1208 нет стадий/воронки; доп. шагов/БП со стороны пайплайна не требуется.
Полный вывод и план записи — в `tasks/phase_4_01_done_button_investigation.md`.

### 4_02 — Сбор значений (дефолт + подтверждение)

`src/fill.py: collect_values(day, cfg, *, interaction, applied_to_all=None) -> Values|SKIP|ABORT`:
- дефолты из `cfg.defaults` (task_description="Общие задачи подразделения", hours=8);
- интерактив: Enter (принять), текст (переопределить), `skip`, `abort`, `all` (применить дефолт
  ко всем); опция «применить ко всем оставшимся» — отдельный y/N после первого ручного ввода;
- `--no-interaction`: дефолты без вопросов; валидация часов (число > 0).

### 4_03 — Создание учёта 1218 (plan→execute)

`src/fill.py: build_payload(...)` (чистая функция) + `create_log(...)` с гардом идемпотентности:
- payload (коды/категория/техкод — из Config): `parentId1208`, `categoryId=63`, договор `T512_2`,
  техкод `"2"`, описание, часы. «Итог» и ручная запись `ufCrm46_1742997115` дня НЕ выполняются;
- **гард `_reread_guard`** перед каждой записью: перечитать день, заново проверить 4-дневное окно
  (по свежей дате Europe/Moscow на момент записи) и пустоту «Работ за день» (fail-closed);
- dry-run → `item_add(plan_only=True)` (показ плана без записи); боевой → `item_add` + возврат ID.

### 4_04 — Верификация

`src/fill.py: verify_log(...)`: перечитывает день (учёт ∈ `ufCrm46_1742997115`) и сам учёт
(`parentId1208`, описание, часы). Расхождения логируются как ошибка. `run_fill(...)` печатает
сводную таблицу: статус `filled`/`skipped`/`error`/`dry-run`/`aborted` + причина/ID по каждому
кандидату.

### Гейт боевой записи (main.py `_cmd_fill`)

Боевая запись возможна ТОЛЬКО при `--confirm-write` И БЕЗ `--dry-run`. По умолчанию и при
`--dry-run` — dry_run=True. Оба флага → dry-run (с предупреждением). Перед боевым режимом — заметный
warning. Добавлена ранняя проверка fill-конфигурации (понятная ошибка вместо KeyError посреди пакета).
