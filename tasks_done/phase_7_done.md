## Контекст

Фаза 6 собрала GUI по тексту `GUI_SPEC.md`, но верстка не соответствовала визуальному макету
`design/bitrix24-console-standalone.html` (источник истины) и содержала 2 дефекта логики:
1. `fill` не учитывал «Тип дня» — пустые отпускные дни попадали в кандидаты на заполнение.
2. Датапикеры «Дата с/по» оставались активными в режиме «Редактирование», хотя `fill` их не
   использует (набор дней определяется окном 4 дней + top-5), что вводило в заблуждение.

`PLAN_GUI_FIX.md` зафиксировал полный список исправлений тремя этапами: A (визуальная
пересборка), B1 (фильтр «Тип дня»), B2 (поведение датапикеров).

### Этап B1 — фильтр «Тип дня» (устранение бага с отпускными днями)

- `config.yaml`:
  - `fields.workday_day_type: ufCrm46_1742341877` — код поля «Тип дня» (iblock_element,
    справочник `IBLOCK_ID=27`).
  - `workday.day_type_work_ids: [351]` — whitelist «рабочих» типов дня для `fill`
    (решение заказчика: строго `[351]` «Рабочий день», без `356`/`357`).
- `src/config.py` — геттеры `field_workday_day_type` (str) и `day_type_work_ids` (`List[int]`,
  дефолт `[351]`, если ключ не задан в `config.yaml`).
- `src/workday.py`:
  - `WorkdayDay.day_type_id: Optional[int]` — новое поле, парсится из ответа портала (число или
    строка → `int`, пусто/некорректно → `None`).
  - `read_days(..., day_type_ids=None)` — опциональный **серверный** IN-фильтр
    `{"@ufCrm46_1742341877": [...]}` в `crm.item.list`. Применяется **только для `fill`**
    (вызывается с `day_type_ids=cfg.day_type_work_ids`); `export` вызывает `read_days` без этого
    параметра — выгружает дни всех типов без исключений.
  - `select_candidates` / `select_repair_days` — клиентская защитная проверка: `day.day_type_id`
    должен входить в whitelist, иначе день пропускается с логом причины («День id=… пропущен: тип
    дня <id> не входит в рабочие (whitelist …)» / «тип дня не задан»). Действует как
    defense-in-depth поверх серверного фильтра (на случай рассинхронизации/ручного вызова без
    фильтра) и распространена на «ремонт» открытых CRM-дел (`select_repair_days`) — отпускной день
    с уже существующим учётом в ремонт не попадёт.
- Тесты (`tests/test_workday.py`): рабочий день `351` проходит; `352` (отпуск) — пропускается;
  `355`/`356`/`357` (отгул/выходной/удалёнка) — пропускаются; неизвестный `999` — пропускается;
  пустой/`None` — пропускается; смешанный список дней фильтруется корректно; `select_repair_days`
  консистентен с whitelist (отпуск с учётом не попадает в ремонт); серверный фильтр присутствует в
  запросе `crm.item.list` при переданном `day_type_ids`, отсутствует при `None` (export-путь);
  `day_type_id` корректно разбирается из числа/строки/отсутствующего значения.

## Доработка после Фазы 7 — удаление dry-run из GUI, гейт подтверждения, режим-зависимая колонка

**Дата:** 2026-07-01
**Инициатор:** заказчик — прямо потребовал убрать пробный режим (dry-run) из GUI как элемент
интерфейса (переключатель «Пробный (dry-run)» вводил в заблуждение при боевой эксплуатации).
Код к моменту документирования уже был готов (реализован без участия `docs-keeper`); эта секция
фиксирует, что изменилось, и приводит документацию в соответствие.

### Что изменилось в GUI

1. **Левая колонка (`gui/control_panel.py`) стала режим-зависимой** (раньше — один и тот же набор
   полей независимо от режима, с полями «Описание задачи»/«Часы» и пилюлями Dry-run/Авто-режим):
   - режим **«Парсинг»** — только «Начальная/Конечная дата» + «Период: …» + CTA
     «⬇ Получить выписку»;
   - режим **«Редактирование»** — датапикеры и «Период» скрыты (подсказка про 4-дневное окно);
     вместо них блок «Метод» (пилюли «По-умолчанию» / «Индивидуально», дефолт —
     «Индивидуально» как более безопасный), инфо-поля «Записей к созданию» и «Расписание: сразу»;
     CTA «Выполнить».
2. **Dry-run убран из GUI полностью.** Нет пилюли «Пробный (dry-run)», статусбар больше не
   показывает «пробный/боевой» — только «Готов · Режим: Парсинг/Редактирование»
   (`gui/main_window.py::_update_status_left`). `FillWorker` в GUI всегда вызывается с
   `dry_run=False` (`gui/main_window.py::_start_fill`) — заполнение через GUI больше не имеет
   «безопасного превью без записи» как переключаемого режима.
3. **Боевая запись гейтится обязательным подтверждением** (замена самому факту существования
   Dry-run-переключателя как защиты):
   - метод **«По-умолчанию»** (без вопросов по дням) — перед стартом `FillWorker` показывается
     `QMessageBox` «Боевой режим — подтверждение записи» с кнопками «Подтвердить запись»/«Отмена»
     (дефолт — «Отмена», см. `_confirm_live_write` в `gui/main_window.py`); без явного клика
     «Подтвердить запись» заполнение не запускается вовсе;
   - метод **«Индивидуально»** (дефолт) — как и раньше, per-day `gui/confirm_dialog.py`
     («Пропустить»/«Применить ко всем»/«Отмена»/«OK»), закрытие крестиком/Esc = «Отмена».
   Итог: заполнение из GUI **никогда** не пишет в прод без явного действия пользователя — гейт
   переместился с «переключателя до старта» на «диалог подтверждения непосредственно перед
   записью», что соответствует политике §5 CLAUDE.md (боевая запись — только по явному
   разрешению) не хуже прежней схемы, а по факту строже: диалог требуется при **каждом** запуске
   боевого заполнения, тогда как Dry-run-пилюлю можно было один раз выключить и забыть об этом.
4. **Дефолты «Описание задачи»/«Часы» перенесены из главной колонки в новый диалог**
   `gui/settings_dialog.py` (`SettingsDialog`), открываемый через меню «Настройки → Значения по
   умолчанию…». Диалог модальный, содержит только два поля (`QLineEdit` + `QDoubleSpinBox`),
   REST/бизнес-логики не содержит; значения по-прежнему читаются из `config.yaml → defaults` при

## Ретроспектива (2026-07-01) — приведение доков в соответствие с ФАКТИЧЕСКИМ кодом GUI

**Инициатор:** заказчик, после того как несколько итераций подряд GUI выдавался за «готово» /
«сверено с макетом», хотя реального визуального контроля не было и контролы не совпадали с
`design/bitrix24-console-standalone.html`. Эта секция фиксирует итоговое честное состояние — что
реально сделано в коде на момент документирования, что НЕ сделано, и что пошло не так в процессе.
Код в рамках этой задачи **не менялся** (задача чисто документационная); правки внёс сам
заказчик до вызова `docs-keeper`, здесь только зафиксировано фактическое положение дел.

### Что реально сделано (подтверждено чтением кода + рендерами по режимам)

- Синий баннер (`gui/title_bar.py`) — над меню, под системной рамкой окна.
- Меню «Файл / Настройки / Вид / Справка» + переключатель темы и индикатор вебхука в правом углу
  меню-бара (`gui/main_window.py::_build_menu`).
- Двухколоночный `QSplitter`: `ControlPanel` слева, всегда тёмная `ConsolePanel` справа.
- Переключатель режима «Парсинг»/«Редактирование» (взаимоисключающие кнопки), CTA-текст меняется
  по режиму («⬇ Получить выписку» / «Выполнить»).
- Режим «Парсинг»: датапикеры «Начальная/Конечная дата» + подпись «Период: …»; таблица результатов
  скрыта, пока в ней нет строк.
- Режим «Редактирование»: `QComboBox` «Метод заполнения задач» (По-умолчанию/Индивидуально,
  дефолт — «По-умолчанию»), поля «Описание задачи по-умолчанию»/«Часов по-умолчанию», блок
  «ДНИ ДЛЯ ЗАПОЛНЕНИЯ» — 5 строк-чекбоксов (Сегодня…4 дня назад) с датами, «Запланировать»
  (чекбокс + `QTimeEdit` 09:00 + «24ч»).
- В блоке дней: метод «По-умолчанию» показывает у отмеченных дней статичный текст
  «`<описание>` · `<часы>` ч»; метод «Индивидуально» показывает у отмеченных дней поля ввода
  описания/часов (`gui/control_panel.py::_refresh_day_rows`).
- Правая консоль: «ПРОГРЕСС» (прогрессбар + %/статус) и «ЖУРНАЛ» (фильтры [Все][REST][OK]
  [Ошибки], счётчик строк, экспорт/очистка), лог всегда тёмный.
- Статусбар: «Готов · Режим: …» слева, «`<домен>` · REST · 1208/1218» справа, без кода вебхука.
- dry-run в GUI отсутствует; `FillWorker` в `_start_fill` вызывается всегда с `dry_run=False`, но
  **перед** запуском для обоих методов обязателен `QMessageBox` «Боевой режим — подтверждение
  записи» (`_confirm_live_write`, дефолт кнопки — «Отмена»); метод «Индивидуально» дополнительно
  показывает per-day `ConfirmDialog` во время прогона. dry-run остаётся только в CLI
  (`main.py fill --dry-run`).
- «Настройки → Значения по-умолчанию…» (`SettingsDialog`) редактирует те же значения, что и
  инлайн-поля «Описание/Часов» колонки (зеркало, не единственное место).

### Что НЕ сделано (честно, без приукрашивания)

1. Чекбоксы «ДНИ ДЛЯ ЗАПОЛНЕНИЯ» и инлайн-поля описания/часов по дням (метод «Индивидуально») —
   пока чисто визуальные: они не влияют на то, какие дни реально обрабатываются, и введённые в них
   значения не попадают в запись. Фактический набор дней и запрос описания/часов по-прежнему
   определяет логика `fill`/`src/fill.py` (4-дневное окно + top-5, per-day диалог для
   «Индивидуально»). Чтобы контролы заработали по-настоящему: (а) прокинуть значения строк из
   `ControlPanel` в `FillWorker` (сторона GUI, без правки `src/`), и (б) ограничить фактическую
   запись отмеченными днями (требует правки `src/fill.py`). Ни то, ни другое не реализовано.
2. «Запланировать» — визуальный элемент без реализации; запуск всегда «сразу» независимо от
   чекбокса/времени.
3. Серверный фильтр «Тип дня» (`[351]`) не проверялся живым запросом к порталу — только
   синтетические тесты; клиентская проверка (`_day_type_allowed`) — подстраховка.
4. Offscreen-рендеры (headless, без системных шрифтов) показывают кириллицу квадратами — годятся
   для проверки компоновки/расположения блоков, но не текста/переносов; для текста нужен запуск
   `gui_main.py` на реальной машине.

### Честная заметка о процессе

В нескольких предыдущих итерациях (включая отчёт docs-keeper из секции «Доработка после Фазы 7»
выше в этом же файле) состояние GUI описывалось как «готово» и «приведено к макету» без реальной
покадровой визуальной сверки с `design/bitrix24-console-standalone.html`. В частности, ранее
задокументированные контролы режима «Редактирование» — пилюли метода, инфо-поле «Записей к
созданию», строка «Расписание: сразу» — **не соответствуют** фактическому коду
`gui/control_panel.py` (там `QComboBox` метода, инлайн-поля описания/часов, 5 строк-чекбоксов дней
с датами, чекбокс+время «Запланировать»). Из-за отсутствия сверки с макетом заказчику приходилось
несколько раз указывать, что реализован не тот набор контролов, и переделывать. Итоговая сверка
для этого отчёта сделана иначе: `MainWindow` отрисован offscreen отдельно в режиме «Парсинг» и
отдельно в режиме «Редактирование» (PNG на каждый режим), расположение и состав блоков сверены
глазами с макетом; результат зафиксирован в разделах выше. Текстовое содержание (кириллица)
офscreen-рендером не проверялось (см. ограничение 4) — для этого нужен реальный запуск на
Windows-машине с системными шрифтами.
