## Что сделано по подфазам

### 6_01 — Каркас окна + вендоренная тема

- `gui/__init__.py` — пакет GUI.
- `gui/theme.py` — вендоренный `ThemeManager` + `PALETTES` + механизм подстановки `@token@` в QSS,
  перенесённый из дизайн-системы `United_Stand_Platform` (USP) без живой зависимости на USP
  (решение из `phase_6_00_overview.md`).
- `resources/styles/main.qss` — вендоренная тема, плейсхолдеры `@token@` подставляются
  `ThemeManager` из палитры.
- `gui/main_window.py` — `QMainWindow`: меню «Файл/Вид/Справка», ribbon, панель параметров,
  таблица результатов + панель деталей, лог-панель, статусбар; оркестрация воркеров;
  `closeEvent` (см. устранённое замечание ревью ниже).
- `gui_main.py` — точка входа: `load_config()` → `setup_logging()` с маскированием секрета →
  `MainWindow` → `theme.apply(system)`.

### 6_02 — Лог, параметры, статусбар

- `gui/log_panel.py` — тёмный `QTextEdit` + `QLoggingHandler`, подключённый к логгеру `"workday"`;
  `SecretMaskingFilter` навешан и на сам хендлер (defense-in-depth для дочерних логгеров

- `gui/params_panel.py` — ввод дат `дд.мм.гггг`, описание/часы, кнопка «Открыть папку» (для
  результата `export`).
- Статусбар интегрирован в `gui/main_window.py`.

### 6_03 — Ribbon, QThread-воркеры, таблица/детали

- `gui/ribbon.py` — кнопки «Выгрузить»/«Заполнить» + переключатели Dry-run (ВКЛ по умолчанию) и
  Авто-режим.
- `gui/worker.py` — `ExportWorker`/`FillWorker` на `QThread`. Интерактив `fill` реализован
  monkeypatch'ем `builtins.input` и перехватом `sys.stdout` (для извлечения id/даты текущего дня
  из приглашения), блокировка диалогом — через `QMutex`/`QWaitCondition` с флагом `_waiting`.
  `src/fill.py` и остальной `src/` **не изменялись** — интерактив встроен снаружи, как
  предписано решением `phase_6_00_overview.md` (§2: "src/ НЕ менять").
- `gui/result_table.py` — таблица результатов в двух режимах (`export`/`fill`), цвет статусов по
  палитре через чистую функцию `status_color_token` (без побочных эффектов, тестируема напрямую).
- `gui/detail_panel.py` — панель деталей дня/результата; `raw`-поля Bitrix24 не выводятся.

### 6_04 — Диалог подтверждения + меню/темы

- `gui/confirm_dialog.py` — `QDialog` подтверждения значений по дню: «Пропустить» / «Применить ко

- Меню «Вид» в `gui/main_window.py` — переключение тем (через вендоренный `ThemeManager`).

### Ревью (`reviewer`, read-only)

Первый проход — блокеров нет. Замечания (Should-fix), все устранены:

1. **`closeEvent` отсутствовал** → добавлен: разблокировка ожидающего потока (если воркер ждёт
   ответа диалога) + корректный `quit()`/`wait()` QThread, без «QThread Destroyed while running»
   и без утечки monkeypatch (`builtins.input` гарантированно восстанавливается).
2. **Паритет пре-чека конфигурации `fill` и предупреждения «БОЕВОЙ РЕЖИМ»** с `main.py` — GUI
   воспроизводит тот же пре-чек и то же предупреждение перед боевой записью, что и CLI.
3. **Подтверждение Yes/No перед боевой АВТО-записью** — если одновременно сняты Dry-run и включён
   Авто-режим (то есть будет выполнена серия боевых записей без диалога по каждому дню), GUI
   запрашивает дополнительное подтверждение «да/нет», прежде чем стартовать воркер.
4. **`SecretMaskingFilter` на самом `QLoggingHandler`** — добавлен (defense-in-depth) в дополнение
   к фильтру на логгере `"workday"`, чтобы дочерние логгеры `workday.*` тоже гарантированно
   маскировались на уровне GUI-хендлера.

Повторное ревью ключевого threading-фикса (`closeEvent` + `QMutex`/`QWaitCondition`-хэндшейк)
пройдено — корректность подтверждена.

**Итоговый вердикт:** approve, блокеров нет.

## Безопасность

- Секрет вебхука не печатается и не попадает в лог-панель / панель деталей / таблицу результатов:
  `raw`-поля Bitrix24 в `DetailPanel`/`ResultTable`/`LogPanel` не выводятся; `setup_logging()` с
  `SecretMaskingFilter` выполняется до любых REST-операций; фильтр продублирован на
  `QLoggingHandler`.
- Dry-run включён по умолчанию (toggle в ribbon стартует в состоянии ВКЛ).
- Гейт боевой записи соблюдён и не ослаблен: `dry_run=False` передаётся в `run_fill` только при
  явно снятой галочке Dry-run в UI — эквивалент CLI-флага `--confirm-write` без `--dry-run`.
- Политика 4 дней не дублируется в GUI-коде — она по-прежнему живёт исключительно в
  `run_fill`/`_reread_guard` (`src/fill.py`, без изменений).
- Qt-потокобезопасность: REST/`fill`-логика выполняется в `QThread`-воркерах, обновление UI —
  только в главном потоке через сигналы/слоты.

## Известное расхождение (не баг, задокументировано намеренно)

GUI резолвит `export.output_dir` **относительно корня проекта**, тогда как CLI (`main.py`)
резолвит тот же путь **относительно текущей рабочей директории (CWD)**. Поведение GUI признано
более надёжным (не зависит от того, откуда запущен процесс), поэтому расхождение оставлено
намеренно, а не унифицировано с CLI. Если в будущем потребуется единообразие — нужно явное
решение заказчика, какое поведение считать эталонным.

## Отложенная верификация (открытый пункт)

Полноценный **боевой end-to-end тест `fill` через GUI** не проводился: на момент закрытия фазы
заказчик в отпуске, и в портале все актуальные дни в окне 4 дней помечены типом «Отпуск»
(столбец «Тип дня» ≠ «Рабочий день»), поэтому подходящих для боевой записи дней нет и не будет
в ближайшие дни. Это не блокирует закрытие фазы: режим по умолчанию — dry-run, вся логика покрыта
тестами и read-only smoke-проверкой окна. **Боевой прогон отложен** до возвращения заказчика и
появления в портале фактических рабочих дней без учёта — фиксируется как пункт для последующей
ручной верификации (не отдельная подфаза, а контрольная точка перед реальной эксплуатацией GUI).
