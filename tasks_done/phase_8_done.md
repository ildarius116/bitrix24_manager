# Отчёт о завершении Фазы 8.01 — «Индивидуально»: выбор дней реально управляет записью + блокировка недоступных

**Дата закрытия:** 2026-07-01
**Исходная спецификация:** `../tasks/phase_8_01_individual_day_selection.md` (сохранена без изменений,
статус переключён на DONE со ссылкой на этот отчёт)
**Статус:** DONE

## Контекст

После Фазы 7 в README честно фиксировалось известное ограничение: чекбоксы «ДНИ ДЛЯ ЗАПОЛНЕНИЯ» и
инлайн-поля описания/часов в методе «Индивидуально» были декоративными — `ControlPanel.selected_days()`
и `per_day_values()` существовали, но нигде не вызывались; реальную запись вёл `run_fill()`, который
сам отбирал топ-5 кандидатов (`select_candidates`) и в интерактиве спрашивал каждый день через
per-day `ConfirmDialog`, игнорируя левую панель. Фаза 8.01 закрывает это ограничение для метода
«Индивидуально» и добавляет блокировку недоступных дней в этом методе.

## Что сделано

### Часть B — выбор/значения левой панели реально управляют записью

- Per-day `ConfirmDialog` **удалён полностью**: файлы `gui/confirm_dialog.py` и
  `tests/test_gui_confirm_dialog.py` удалены из дерева. Мёртвая проводка (`confirm_requested` →
  `_on_confirm_requested`, блокирующее ожидание) убрана из `gui/main_window.py`.
- Механизм — существующий синхронный перехват `builtins.input` в `FillWorker` (`gui/worker.py`),
  переработан под безальтернативный (без диалогов/блокировки потока) режим:
  - `FillWorker.__init__` принимает и хранит `selected_days: Optional[set[date]]` и
    `per_day_values: dict[date, (str, float)]` (по умолчанию `None`/`{}` — не ломает метод
    «По-умолчанию» и `export`, которые их не передают).
  - `_StdoutTap` продолжает наполнять `self._last_day_date` строкой `dd.mm.yyyy` из заголовка
    дня, который печатает `run_fill`/`collect_values`.
  - `_fake_input(prompt)` — синхронная функция без диалогов:
    - парсит `_last_day_date` в `date` (`_parse_last_day_date`);
    - если `selected_days is None` (метод «По-умолчанию») — прежнее поведение (дефолтные
      описание/часы, «N» на «применить ко всем»);
    - если `selected_days` задан (метод «Индивидуально»): дата не входит в `selected_days` →
      промпт «Описание» получает `"skip"` (день пропускается в `collect_values`); дата входит —
      выставляется `self._current_hours` из инлайн-часов дня и возвращается инлайн-описание из
      `per_day_values`; промпт «Количество часов» отдаёт `self._current_hours`; промпт «ко всем»
      всегда «N» (нет режима «применить ко всем» без диалога).
  - `MainWindow._start_fill` при методе «Индивидуально» собирает `selected_days`/`per_day_values`
    из `ControlPanel` и передаёт в `FillWorker`; `provide_confirmation` оставлен как no-op (его
    по-прежнему дёргает `closeEvent`, чтобы не блокировать закрытие окна на старом коде).
  - `src/` не изменён — вся логика перехвата осталась в `gui/worker.py`; `_reread_guard`
    (`src/fill.py`) остаётся страховкой окна/пустоты и не дублируется в GUI.

### Часть A — блокировка недоступных чекбоксов в «Индивидуально»

- `gui/worker.py`:
  - `classify_day_states(days, cfg, today) -> dict[str, dict]` — чистая функция без REST; ключ —
    ISO-дата, значение `{"fillable": bool, "reason": str}`. Причины: «нет записи «Рабочий день»»
    (даты нет среди прочитанных `WorkdayDay`), «не рабочий день» (тип дня не проходит whitelist
    `cfg.day_type_work_ids`), «вне окна 4 дней», «уже заполнен (N)» (есть привязанные учёты),
    иначе `fillable=True`. Окно дат — те же 5 строк, что рисует панель
    (`today − edit_window_days … today`).
  - `DayStatesWorker(QThread)` — read-only: `B24(config)` → smoke → `read_days(...,
    day_type_ids=cfg.day_type_work_ids)` → `classify_day_states`. Сигналы `states_ready(dict)` /
    `states_failed()`; в прод не пишет, виджеты не трогает (только сигналы, апдейт UI — в
    main thread через `MainWindow`).
- `gui/control_panel.py`:
  - Сигнал `day_states_requested`; кнопка **«↻ Обновить»** в шапке секции «ДНИ ДЛЯ ЗАПОЛНЕНИЯ» —
    видна только в методе «Индивидуально».
  - `set_day_states_loading()` — чекбоксы disabled + подпись «загрузка…» на время запроса.
  - `apply_day_states(mapping)` — по дате строки: `fillable=True` → чекбокс enabled; иначе →
    disabled, `setChecked(False)`, подпись/tooltip с причиной. Применяется только в методе
    «Индивидуально»; при возврате к методу «По-умолчанию» блокировка снимается (все enabled).
  - `apply_day_states_error()` — фолбэк при ошибке загрузки: чекбоксы возвращаются в активное
    состояние, подпись «не удалось получить статусы».
  - `day_states_requested` эмитится при переключении на метод «Индивидуально» (в режиме
    «Редактирование») и по кнопке «Обновить».
- `gui/main_window.py`: владеет `DayStatesWorker` (`self._day_states_worker`, тот же паттерн, что
  и `SmokeWorker`); guard от повторного запуска, пока предыдущий не завершился, и от запуска во
  время активной операции export/fill; `states_ready → control.apply_day_states`,
  `states_failed → control.apply_day_states_error`.

### Общий диалог-предупреждение боевой записи

`_confirm_live_write` (`gui/main_window.py`) — не тронут, остаётся безусловным гейтом (`QMessageBox`
«Боевой режим — подтверждение записи») для обоих методов заполнения, до старта `FillWorker`.

### Что НЕ менялось

- Метод «По-умолчанию» — поведение прежнее (`selected_days=None`, дефолтное описание/часы,
  никакой блокировки чекбоксов).
- `export` — не затронут.
- `src/` — не изменён (по требованию спеки).

## Известные ограничения (перенесены из спеки, актуальны)

1. **«Ремонт» закрытия дел** (`complete_day_activities` в `run_fill`) идёт по уже заполненным дням
   окна независимо от галок в панели — учёт при этом НЕ создаётся и не перезаписывается, только
   закрывается висящее CRM-дело «Выполнено» существующего учёта. Прикрыт общим диалогом-
   предупреждением боевой записи (`_confirm_live_write`), но в разрезе «выбор дней управляет
   записью» это не совсем так для уже готовых дней — их дело закроется независимо от чекбокса.
2. **Отмеченный день вне топ-5 кандидатов `run_fill` не запишется.** Отбор кандидатов
   (`select_candidates`) остаётся источником итогового набора дней, которые вообще обходит
   `collect_values`; выбор в панели может только СУЗИТЬ этот набор (пропустить неотмеченные), но
   не расширить его сверх топ-5. На практике окно ≤5 дней совпадает с топ-5, поэтому расхождения
   не возникает; страховка — `_reread_guard` (`src/fill.py`), не дублируется в GUI.
3. **Полночная рассинхронизация (staleness).** Строки дней в панели строятся на `today_moscow()`
   в момент конструирования/показа панели, а `DayStatesWorker` читает статусы на `today` в момент
   запроса запроса пользователем («Обновить»/вход в метод). Если окно GUI остаётся открытым через
   полночь без переоткрытия, возможен сдвиг на день между отображаемыми датами строк и датами,
   на которые реально проверялась/будет применена доступность. Это пред-существующее свойство
   (панель и раньше не пересчитывала даты на лету) — в рамках Фазы 8.01 не чинилось.

Все три пункта продублированы в `README.md` (раздел «Графический интерфейс (GUI)» →
«Известные ограничения GUI»).

## Доработка (2026-07-01, тот же день) — disabled «Запланировать» в «Индивидуально» + косметика кнопки «Обновить»

После первичного документирования фазы внесены две небольшие правки в код `gui/`; код к моменту
документирования уже был готов (эта секция только приводит доки в соответствие).

1. **Строка «Запланировать» (чекбокс + `QTimeEdit` + подпись «24ч») теперь disabled в методе
   «Индивидуально».** Реализовано `gui/control_panel.py::_sync_schedule_enabled()` — вызывается из
   `_apply_mode`/`_on_method_changed` при каждой смене метода. Активна строка только в методе
   «По-умолчанию» (где, как и раньше, ни на что не влияет — «Запланировать» остаётся визуальным
   элементом без реализации планирования, см. известное ограничение выше). В методе «Индивидуально»
   строка полностью недоступна, включая вложенный `QTimeEdit` (доступен только когда родительская
   строка активна И чекбокс отмечен).
2. **Кнопка «↻ Обновить» окрашена в акцентный синий цвет** — чисто косметическая правка, поведение
   не изменилось. Добавлен селектор `QPushButton#RefreshDaysButton` в
   `resources/styles/main.qss` (фон `@accent@`, белый текст, отдельные стили `:hover`/`:pressed`/
   `:disabled`).
